What to actually learn from the math sections
· not what the syllabus says — what you need to carry into every algorithm
Linear Algebra · Days 1–8
Vectors & Matrix operations
Learn matrix multiplication as function composition, not row-times-column arithmetic. Know why Ax = b has 0, 1 or ∞ solutions.
→ Every layer of a neural net is a matrix multiply. Understanding this visually lets you debug shapes instantly.
Projections & Least squares
Derive P = A(AᵀA)⁻¹Aᵀ yourself. Understand that least squares is projecting y onto the column space of X.
→ Linear regression, PCA, attention — all fundamentally projections. If you get this, you get all three.
Eigenvalues & SVD
Learn SVD as three geometric operations: rotate → scale → rotate. Know Eckart-Young (best rank-k approximation). Know that PCA eigenvectors = right singular vectors of centred X.
→ PCA, dimensionality reduction, noise filtering, recommendation systems. SVD is everywhere.
What to actually memorise
Don't memorise determinant formulas. DO memorise: (1) four fundamental subspaces, (2) rank-nullity theorem, (3) projection matrix formula, (4) SVD A = UΣVᵀ, (5) PCA = eigen(XᵀX).
→ These five things appear constantly. Everything else you can re-derive.
Probability & Statistics · Days 9–11, 15–16
Bayes theorem — deeply
Don't just know the formula. Know why posterior = prior × likelihood / evidence. Know the difference between MAP (point estimate) and full posterior (distribution).
→ MAP estimate = L2-regularised MLE. Bayesian models = full posterior. This distinction matters for every regularisation decision.
MLE — derive it for 3 distributions
Gaussian MLE → gives you μ̂ and σ̂². Bernoulli MLE → gives you cross-entropy loss. Poisson MLE → gives you count regression. Do all three on paper.
→ Every loss function in ML is a negative log-likelihood. Understanding this removes all mysticism from loss choice.
KL divergence & entropy
Know why KL(P‖Q) ≠ KL(Q‖P). Know that cross-entropy = KL + H(P). Understand information gain = entropy reduction.
→ VAE loss, decision tree splits, knowledge distillation, diffusion models — all use these. One mental model covers all.
What to skip
Don't spend days on CDF tables or hypothesis test lookup tables. You'll use scipy for that. Focus on intuition: what is a p-value really saying, what does power mean geometrically.
→ You need the concepts for A/B test design decisions, not to hand-compute test statistics.
Optimisation · Days 12–14
Gradient descent — the geometry
Understand why we subtract the gradient (it points uphill, we want downhill). Know that SGD with noise can escape local minima that full-batch gradient descent cannot. Understand learning rate as step size, not a magic number.
→ Every training loop you ever write depends on this intuition. Debugging loss curves requires it.
Adam — derive it once, understand it forever
Adam = momentum (m̂) + per-parameter learning rates (v̂). Bias correction exists because m and v start at zero, making early estimates too small. Know defaults: β₁=0.9, β₂=0.999, ε=1e-8 and why they are those values.
→ You'll use Adam in 90% of DL projects. Knowing what it's doing lets you tune it when training is unstable.
Convexity — just enough
Know what a convex function is (bowl shape, any local min = global min). Know that logistic regression loss is convex but neural nets are not. Know why that matters — and why we train neural nets anyway.
→ Lets you reason about when gradient descent is guaranteed to work versus when it might fail.
Matrix calculus — practical focus
Learn numerator layout convention. Know ∂(Wx)/∂W = xᵀ, ∂(xᵀAx)/∂x = 2Ax. Practice deriving the gradient for a 2-layer net by hand — once. After that, PyTorch autograd does it.
→ Doing backprop once by hand builds permanent intuition. You don't need to do it every time.

How to actually learn ML — what to do at each stage
The learning loop (per topic)
1. Watch — don't take notes
First pass of any video: just watch. Resist the urge to pause and note everything. You're building a mental scaffold, not copying a textbook. Note only things that surprised you.
2. Derive on paper — before code
After every video, close the screen and rederive the key equation from memory on paper. For linear regression, write the normal equations. For backprop, draw the computation graph. This is the hardest step and the most important one.
3. Implement from scratch — no copy-paste
Write from_scratch.py with the algorithm doc closed. Use numpy, not sklearn. When you hit a shape error, that's the learning. Compare output to sklearn's version — same numbers means correct.
4. Explain it out loud
Set a timer for 90 seconds and explain the algorithm to nobody. If you stumble on the motivation ("why does this work?"), that's the gap to fill. You don't understand something until you can explain it simply.
5. Read the paper — after implementation
Read the original paper after you've implemented, not before. Your implementation gives you a question to bring to the paper. Without it, papers feel abstract. With it, they feel like answers.
Common traps to avoid
Passive watching is not studying
3B1B and StatQuest are excellent but watching them without implementing is like watching someone do push-ups. The video is the map — the code is the territory. If you haven't opened a text editor, you haven't studied yet.
Don't get stuck on proofs forever
For each topic, there are proofs you must do once (SVD derivation, MLE for Gaussian, Bellman optimality), and proofs you can read and move on (convergence rates, Mercer's theorem). Know the difference. Spending 3 days on a convergence proof instead of implementing the algorithm is a trap.
sklearn first is backwards
Most people learn sklearn.RandomForestClassifier() before they know what a tree is. This roadmap correctly inverts this — always implement from scratch first. Once you do, sklearn becomes a verified reference implementation, not a black box.
Don't collect resources — use one per topic
Your plan already has good videos for everything. Don't spend the day of Day 5 finding "the best" eigenvalue video. Use what's listed, implement, move on. Resource-hunting is procrastination in disguise.
Treat review days seriously
Days 63–68 look easy on the plan. They're not. The goal is to rederive every major result without notes. If you can't, that's a real gap — go back. These days are the difference between "studied ML" and "knows ML".
What builds actual skill
Build a project that breaks
Apply every algorithm to a real dataset (Kaggle, UCI, your own data). When your model underperforms, diagnose it: is it the model, the features, the hyperparameters, the evaluation? Diagnosing failures teaches more than studying successes.
Connect algorithms across phases
When you study VAE (Day 55), connect it backwards to KL divergence (Day 10) and MLE (Day 11). When you study PPO (Day 75), connect the clipping to TRPO and to the trust region concept. Your plan has these connections implicit — make them explicit in your notes.
Interview yourself daily
Before ending each day: "What would an interviewer ask about what I learned today?" Answer it. This is not about interview prep — it's a forcing function to identify what you understood versus what you only saw. Common questions: "Why does X work?", "What are the assumptions?", "When does it fail?"
The ratio: 30% theory, 70% code
A rough split that works: 30% of your time understanding why, 70% of your time implementing and debugging. If you spend more than 2 hours on a paper without writing code, you're off-balance. Papers inform code. Code tests understanding of papers.
Write, don't just run
After each topic, write a short (half a page) explanation as if for a blog post. This is more effective than flashcards for ML — algorithms have structure that prose can preserve but cards cannot. Your flash_cards.md on Day 67 should be the result of 67 days of short writings.
